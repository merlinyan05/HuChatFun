# Claude Code 记忆维护指南（给自己看的）

## 核心原则

Claude Code 没有跨会话记忆。它的"记忆"全靠每次启动时读文件。
所以：**你维护文件的质量 = Claude Code 的记忆质量**。

## 你的项目要维护的文件

### 1. `CLAUDE.md`（最重要，自动读取）

放在项目根目录。Claude Code 每次启动自动读。

**必须包含的内容**：

```markdown
## 项目是什么          ← 一两句话说清楚
## 技术路线            ← 模型、框架、硬件
## 目录结构            ← 简版 tree
## 当前进度            ← checklist，标出"当前在这里"
## 关键设计决策        ← 已经确定不再讨论的事
## 上次会话摘要        ← 做了什么、产出了什么、下一步是什么
```

**每次会话结束前说这句话**：

```
把这次会话的进度和关键决策更新到 CLAUDE.md，
包括：完成了什么、产出了哪些文件、下一步做什么。
```

这是最关键的操作。不说这句话，下次开就会失忆。

### 2. `README.md`（辅助，需要主动让它看）

项目说明、快速开始、目录说明。Claude Code 不会自动读，但你可以说"先看一下 README"。

### 3. `docs/` 下的文档

详细的设计文档。CLAUDE.md 里放摘要和引用，详细内容放 docs/ 下。
Claude Code 需要时可以说"看一下 docs/cleaning_strategy.md"。

## 实际操作流程

### 开一个新会话时

```
# 方式一：什么都不说，它会自动读 CLAUDE.md
# 然后直接说你要做的事

继续做阶段2的数据切分

# 方式二：如果它看起来有点懵，让它先读文档
先看一下 CLAUDE.md 和 docs/cleaning_workflow.md，然后继续做阶段2
```

### 会话进行中

正常 vibe coding，该写代码写代码，该调试调试。

中途如果做了重要决策（比如"INC 文件决定跳过不处理"），可以随时说：

```
把"INC 文件跳过不处理"这个决策加到 CLAUDE.md 的关键设计决策里
```

### 关闭会话前（最重要！）

```
更新 CLAUDE.md：
- 当前进度更新到阶段2完成
- 上次会话摘要写清楚做了什么
- 如果有新的坑点或决策也加上
```

## CLAUDE.md 的反模式（不要这样写）

❌ **太长**：超过 200 行就太长了，Claude Code 启动时读太多反而抓不住重点。
把详细内容放 docs/ 下，CLAUDE.md 里只放摘要。

❌ **太模糊**：
```markdown
## 进度
正在做数据处理   ← 这没用，不知道做到哪了
```
✅ 应该写：
```markdown
## 进度
- [x] 阶段1 粗切完成，产出 data/step1_cleaned/（387 个文件）
- [ ] 阶段2 切分 ← 当前在这里，step2_segment.py 写了一半，说话人正则已调好
```

❌ **忘记更新**：CLAUDE.md 里写的还是两周前的进度。
那 Claude Code 下次启动就会以为你还在两周前的状态。

## 这个项目特别需要记住的东西

对于 HuChatFun 这种数据处理 + 训练项目，CLAUDE.md 里最关键的是：

1. **当前在哪个阶段**（step 几）
2. **各阶段数据的统计数字**（处理了多少文件、产出多少条、过滤后剩多少）
3. **语料格式的探查结论**（正则怎么写取决于这个）
4. **踩过的坑**（比如某个正则会误删观点段落）

这些信息每次都更新到 CLAUDE.md，Claude Code 就能在任何时候"接上"你的工作。

## 总结

```
开始会话 → Claude Code 自动读 CLAUDE.md → 它知道项目背景和当前进度
  ↓
工作中 → 重要决策随时写入 CLAUDE.md
  ↓
结束前 → 说"更新 CLAUDE.md" → 它把进度和摘要写好
  ↓
下次开 → Claude Code 读到最新的 CLAUDE.md → 无缝继续
```

就这么简单。核心就一件事：**每次关会话前让它更新 CLAUDE.md**。
